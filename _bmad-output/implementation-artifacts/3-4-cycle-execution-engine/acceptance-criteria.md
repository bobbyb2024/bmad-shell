# Acceptance Criteria

1. **AC1: Ordered Step Execution** — Given a cycle with multiple ordered steps, when the cycle engine executes, then steps run strictly in the configured order (e.g., step 1, then 2, then 3).
2. **AC2: Step Type Logic** — Given a cycle with `generative` and `validation` step types, when the cycle runs for the first time (repetition index 0), then both types execute. On subsequent repetitions (index > 0), then only `validation` steps execute; `generative` steps are skipped.
3. **AC3: Cycle Repetition** — Given a cycle with `repeat: N`, when the engine executes, then it runs N total iterations. The first iteration (index 0) executes both generative and validation steps per AC2. Iterations 1 through N-1 execute only validation steps. If `repeat <= 0`, the engine MUST emit `ErrorOccurred` and skip execution. Note: this is a runtime guard — even if `CycleConfig` schema validation catches this at parse time, the engine must handle it defensively.
4. **AC4: Step Pauses** — Given `pause_between_steps` is configured, when a step completes (and it is NOT the last step of the cycle), then the engine pauses for that duration before starting the next step.
5. **AC5: Cycle Pauses** — Given `pause_between_cycles` is configured, when a cycle repetition completes (and it is NOT the final repetition), then the engine pauses for that duration before starting the next repetition.
6. **AC6: Event Emission** — Given the cycle engine executing, it MUST emit the following events via `EventEmitter`:
   - `CycleStarted(cycle_number: int, provider_name: str)` at start of each repetition, AFTER upfront validation passes (AC11, AC12). `provider_name` is from the provider of the first step in the step list (index 0). `cycle_number` is 1-indexed (1 to `repeat`).
   - `StepStarted(step_name: str, step_index: int)` before each step. `step_name` is `{step.skill}_{step_index}`.
   - `StepCompleted(step_name: str, step_index: int, success: bool)` after each step.
   - `CycleCompleted(cycle_number: int, provider_name: str, success: bool)` after each repetition.
   - `ErrorOccurred(error_type: str, message: str, source: str, recoverable: bool)` on any error. `source` MUST be the `cycle_id` for cycle-level errors or `step_name` for step-level errors.
   - `EscalationChanged(step_name: str, level: str)` if a step output contains an explicit escalation trigger.
7. **AC7: Atomic State Persistence** — Given a step completes, the engine MUST call `StateManager.record_step(state, cycle_id, step_record)` to update the in-memory state, then `StateManager.save(state, path)` to persist atomically, before proceeding to the next step or pause.
8. **AC8: Prompt Resolution** — Given a step with a prompt template, the engine MUST use `TemplateResolver.resolve(prompt, context, step_name=step_name)` to resolve all variables before execution. If resolution fails, it must emit `ErrorOccurred(source=step_name)`, record a failure `StepRecord` with the error message as "outcome", and follow AC10.
9. **AC9: Logging Context** — Given a step execution, the engine MUST use `structlog.contextvars.bind_contextvars()` to attach `step_name`, `provider_name`, and `cycle_id`. Step-level context MUST be removed via `unbind_contextvars("step_name", "provider_name")` in a `finally` block. Cycle-level context MUST be removed via `unbind_contextvars("cycle_id")` (NOT `clear_contextvars()`) in an outer `finally` block to preserve global context.
10. **AC10: Step Failure Handling** — Given a step that fails, the engine MUST emit an `ErrorOccurred` event, record the failure in `StepRecord` (with `outcome=StepOutcome("failure")`), emit `CycleCompleted(success=False)`, persist state, and return the `RunState`. The caller determines failure by inspecting the last `CycleRecord` in `run_history` (not via a `status` field — `RunState` has no `status` field).
11. **AC11: Empty/No-Op Cycle Handling** — Given a cycle with zero steps, the engine MUST emit `ErrorOccurred` and skip execution. Additionally, given a cycle with only `generative` steps and `repeat > 1`, the engine MUST detect this condition **upfront** (before execution begins), emit `ErrorOccurred(error_type="ConfigError", message="Cycle has only generative steps but repeat > 1; iterations 1+ would be no-ops", source=cycle_id, recoverable=False)`, and skip execution entirely.
12. **AC12: Provider Validation** — Given a step whose `provider` int key does not exist in `OrchestratorConfig.providers`, or the provider exists but has an empty/missing `name` (falsy value), the engine MUST emit `ErrorOccurred(error_type="ConfigError", source=step_name, recoverable=False)` and halt the cycle following the AC10 failure protocol. Provider validation MUST occur upfront for all steps before the first `CycleStarted` event.
